Критерии

Требования соответствуют официальному руководству по стилю для Vue-специфичного кода. Более подробно их можно изучить здесь.

  • Б1. Имена компонентов состоят из нескольких слов

    Имена компонентов должны всегда состоять из нескольких слов, за исключением корневого компонента App и встроенных компонентов самого Vue, например, <transition> или <component>.

    Это предотвращает конфликты с существующими или будущими HTML-элементами, поскольку все HTML-элементы именуются одним словом.

    Плохо:

    Vue.component('todo', {
  // ...
})
export default {
  name: 'Todo'
  // ...
}

    Хорошо:

    Vue.component('todo-item', {
  // ...
})
export default {
  name: 'TodoItem'
  // ...
}

    Vue Style Guide

  • Б2. Свойство data компонента должно быть функцией

    При использовании свойства data в компоненте (т. е. везде, за исключением new Vue), значением всегда должна быть функция, возвращающая объект.

    Если использовать объект вместо функции, то data будет общим объектом для всех экземпляров данного компонента. Функция, которая возвращает объект, создаёт собственный уникальный объект данных для каждого экземпляра.

    Плохо:

    Vue.component('some-comp', {
  data: {
    foo: 'bar'
  }
})

// В .vue-файле
export default {
  data: {
    foo: 'bar'
  }
}

    Хорошо:

    Vue.component('some-comp', {
  data: function() {
    return {
      foo: 'bar'
    }
  }
})

// В .vue-файле
export default {
  data() {
    return {
      foo: 'bar'
    }
  }
}

// Также допустим сокращённый вариант со стрелочной функцией
export default {
  data: () => ({
    foo: 'bar'
  })
}

    Vue Style Guide

  • Б3. Входные параметры должны быть определены как можно более подробно

    Определение входных параметров всегда должно быть максимально подробным. В курсе мы обязательно будем указывать type и одно из двух: required (если true) или default (если required: false).

    Плохо:

    // Этого достаточно лишь для прототипа
props: ['status']

    Хорошо:

    props: {
  status: {
    type: String,
    default: 'Hello World'
  },

  level: {
    type: Number,
    required: true
  }
}

    Дефолтное значение для объектов и массивов должно возвращаться из функции.

    Vue Style Guide

  • Б4. Уникальные ключи для v-for

    key с v-for всегда обязателен для компонентов, для поддержания внутреннего состояния компонента и его поддерева.

    Не рекомендуется использовать индекс из цикла. Исключение — статические списки, которые не меняются после инициализации.

    Vue Style Guide

  • Б5. Запрещено использовать v-if с v-for на одном и том же элементе

    Vue Style Guide

  • Б6. Локальные стили компонента

    Для приложений стили в корневом компоненте App и в компонентах шаблона (лейаута) могут быть глобальными, но во всех остальных компонентах должны быть локальными. Есть различные варианты ограничения области действия стилей. Подробнее в стайлгайде. В курсе мы используем БЭМ-стратегию.

    Vue Style Guide

  • Б7. Именование приватных свойств

    Используйте область видимости модуля, чтобы приватные функции были недоступны извне. Если это невозможно, всегда используйте префикс $_ для пользовательских приватных свойств в плагине, примеси и т. п. Таким образом, они не будут рассматриваться как публичный API-интерфейс. Затем, чтобы избежать конфликтов с кодом других авторов, также включайте именованную область (например, $_yourPluginName_).

    Vue Style Guide

  • Б8. Каждый компонент должен быть в отдельном файле

    Это поможет вам быстрее найти компонент, когда потребуется его отредактировать или просмотреть, как его использовать.

    Vue Style Guide

  • Б9. Именование однофайловых компонентов

    Имена файлов однофайловых компонентов должны быть всегда в PascalCase и совпадать с именем компонента.

    Плохо:

    components/
|- mycomponent.vue

components/
|- myComponent.vue

    Хорошо:

    components/
|- MyComponent.vue

components/
|- my-component.vue

    Vue Style Guide

  • Б10. Именование базовых компонентов

    Базовые компоненты (известные как «презентационные», «глупые» или «чистые» компоненты), которые применяют специфичные для вашего приложения стили или соглашения, должны начинаться с определённого префикса, такого, как Base, App или V.

    Эти компоненты закладывают основу для консистентности стилей и поведения в вашем приложении. Они могут содержать только:

    • HTML-элементы;
    • другие базовые компоненты;
    • сторонние UI-компоненты.

    Но они никогда не содержат глобальное состояние (например, из хранилища Vuex).

    Плохо:

    components/
|- MyButton.vue
|- VueTable.vue
|- Icon.vue

    Хорошо:

    components/
|- BaseButton.vue
|- BaseTable.vue
|- BaseIcon.vue

components/
|- AppButton.vue
|- AppTable.vue
|- AppIcon.vue

components/
|- VButton.vue
|- VTable.vue
|- VIcon.vue

    Vue Style Guide

  • Б11. Именование компонентов, используемых в единственном экземпляре

    Компоненты, которые должны иметь только один активный экземпляр, следует начинать именовать с префикса The, обозначая таким образом, что он может быть только один.

    Это не означает, что компонент используется только на одной странице, но означает, что он будет использоваться только один раз на странице. Эти компоненты никогда не принимают каких-либо входных параметров, поскольку они специфичны для вашего приложения, а не их контекста в вашем приложении. Если вы обнаружите необходимость добавления входных параметров, это хороший признак того, что на самом деле этот компонент — для многократного использования, который используется только один раз на странице в данный момент.

    Vue Style Guide

  • Б12. Именование тесно связанных компонентов

    Дочерние компоненты, тесно связанные с родителями, должны включать имя родительского компонента в качестве префикса.

    Плохо:

    components/
|- TodoList.vue
|- TodoItem.vue
|- TodoButton.vue

components/
|- SearchSidebar.vue
|- NavigationForSearchSidebar.vue

    Хорошо:

    components/
|- TodoList.vue
|- TodoListItem.vue
|- TodoListItemButton.vue

components/
|- SearchSidebar.vue
|- SearchSidebarNavigation.vue

    Vue Style Guide

  • Б13. Порядок слов в именах компонентов

    Компоненты должны именоваться с высшего уровня (часто наиболее общих слов) и заканчиваться описательными дополняющими словами.

    Пример:

    SearchButtonClear.vue
SearchButtonRun.vue
SearchInputQuery.vue

    Vue Style Guide

  • Б14. Самозакрывающиеся теги компонентов

    Компоненты без содержимого должны быть самозакрывающимися тегами в однофайловых компонентах, строковых шаблонах и JSX, но никогда в DOM-шаблонах.

    Vue Style Guide

  • Б15. Использование полных слов при именовании компонентов

    Имена компонентов должны состоять из полных слов, а не аббревиатур.

    Исключение: распространённые аббревиатуры. Например: btn. В этом случае допускается название AppBtn.

    Vue Style Guide

  • Б16. Стиль именования входных параметров

    Входные параметры должны всегда использовать camelCase при определении, но kebab-case в шаблонах и JSX. Для JavaScript использовать camelCase является более естественным. Для HTML — kebab-case.

    Плохо:

    props: {
  'greeting-text': String
}

    <WelcomeMessage greetingText="hi"/>

    Хорошо:

    props: {
  greetingText: String
}

    <WelcomeMessage greeting-text="hi"/>

    Vue Style Guide

  • Б17. Элементы с несколькими атрибутами

    Элементы с несколькими атрибутами должны располагаться на нескольких строках, по одному атрибуту на строку.

    Vue Style Guide

  • Б18. Простые выражения в шаблонах

    Шаблоны компонентов должны содержать только простые выражения, а более комплексные должны быть вынесены в вычисляемые свойства или методы.

    Vue Style Guide

  • Б19. Простые вычисляемые свойства

    Комплексные вычисляемые свойства должны быть разделены на максимально простые свойства.

    Vue Style Guide

  • Б20. Значения атрибутов в кавычках

    Непустые значения HTML-атрибутов должны быть обрамлены кавычками (одинарными или двойными, в зависимости от того, что не используется в JS).

    Vue Style Guide

  • Б21. Сокращённая запись директив

    Сокращённую запись директив (: для v-bind:, @ для v-on: и # для v-slot) следует использовать всегда или никогда. В курсе мы используем сокращённую запись.

    Плохо:

    <input v-bind:value="newTodoText" :placeholder="newTodoInstructions" >

<input
  v-on:input="onInput"
  @focus="onFocus"
>

    Хорошо:

    <input :value="newTodoText" :placeholder="newTodoInstructions" >

<input
  @input="onInput"
  @focus="onFocus"
>

    Vue Style Guide

  • Б22. Для управления состоянием используется Vuex

    Vuex должен быть предпочтительным способом для глобального управления состоянием приложения вместо использования this.$root или глобальной шины событий.

  • Б23. Для навигации внутри приложения используется Vue Router

    Навигация внутри приложения должна быть реализована с помощью компонента <router-link>, компонента и методов Vue-роутера.

  • Б24. Линтер не должен выдавать ошибок

    По тем настройкам, которые указаны в курсе.

  • Б25. Расположение элементов интерфейса должно соответствовать вёрстке

    Расположение элементов в выполненном проекте соответствует расположению элементов в начальной вёрстке.

  • Д1. Стиль именования компонентов в JS/JSX

    Стиль именования компонентов в JS/JSX всегда должен быть PascalCase, хотя они могут быть в kebab-case внутри строк для простых приложений, которые используют только глобальную регистрацию компонентов через Vue.component.

    Vue Style Guide

  • Д2. Порядок опций компонента/экземпляра

    Опции компонента/экземпляра должны быть упорядочены консистентно.

    Это порядок по умолчанию, который мы рекомендуем для опций компонентов. Они разделены на категории, поэтому вы поймёте, где добавлять новые свойства из плагинов.

    1. Побочные эффекты (вызывает эффекты вне компонента):
      • el.
    2. Глобальная осведомлённость (требует знаний вне компонента):
      • name;
      • parent.
    3. Тип компонента (изменяет тип компонента):
      • functional.
    4. Модификаторы шаблона (изменение способа компиляции шаблонов):
      • delimiters;
      • comments.
    5. Зависимости шаблона (ресурсы, используемые в шаблоне):
      • components;
      • directives;
      • filters.
    6. Композиция (объединение свойств в опциях):
      • extends;
      • mixins.
    7. Интерфейс (интерфейс компонента):
      • inheritAttrs;
      • model;
      • props/propsData.
    8. Локальное состояние (локальные реактивные свойства):
      • data;
      • computed.
    9. События (коллбэки, вызываемые реактивными событиями):
      • watch;
      • События хуков жизненного цикла (в порядке их вызова):
        • beforeCreate;
        • created;
        • beforeMount;
        • mounted;
        • beforeUpdate;
        • updated;
        • activated;
        • deactivated;
        • beforeDestroy;
        • destroyed.
    10. Нереактивные свойства (свойства экземпляра, независимые от системы реактивности):
      • methods.
    11. Отрисовка (декларативное описание вывода компонента):
      • template/render;
      • renderError.

    Vue Style Guide

  • Д3. Порядок атрибутов элемента

    Атрибуты элементов (в том числе компонентов) должны быть упорядочены консистентно.

    1. Определение (предоставляет параметры компонента):
      • is.
    2. Отображение списка (создаёт несколько вариантов одного элемента):
      • v-for.
    3. Условия (указывается, отрисовывается/отображается ли элемент):
      • v-if;
      • v-else-if;
      • v-else;
      • v-show;
      • v-cloak.
    4. Модификаторы отрисовки (изменяют способ отрисовки элемента):
      • v-pre;
      • v-once.
    5. Глобальная осведомлённость (требует знаний вне компонента):
      • id.
    6. Уникальные атрибуты (атрибуты, требующие уникальных значений):
      • ref;
      • key.
    7. Двусторонняя привязка (объединение привязки и событий):
      • v-model.
    8. Другие атрибуты (все неуказанные связанные или несвязанные атрибуты)
    9. События (обработчики событий компонента):
      • v-on.
    10. Содержимое (перезаписывает содержимое элемента):
      • v-html;
      • v-text.

    Vue Style Guide

  • Д4. Пустые строки между опций компонента/экземпляра

    Необходимо добавить одну пустую строку между многострочными свойствами.

    Хорошо:

    props: {
  value: {
    type: String,
    required: true
  },

  focused: {
    type: Boolean,
    default: true
  },

  label: {
    type: String,
    default: ‘Default label’
  }
},

computed: {
  formattedValue: function () {
    // ...
  },

  inputClasses: function () {
    // ...
  }
}

    Когда компоненты кажутся неразборчивыми и становятся трудными для чтения, то добавление пустых строк между многострочными свойствами может облегчить их беглое изучение во время просмотра.

    Vue Style Guide

  • Д5. Порядок секций в однофайловых компонентах

    Однофайловые компоненты должны всегда использовать один порядок для корневых тегов секций <template>, <script> и <style>, либо <script>, <template> и <style>, заканчиваясь <style>, потому что всегда требуется хотя бы одна из двух других.

    Vue Style Guide

  • Д6.v-if/v-else-if/v-else использовать с key

    Необходимо использовать key вместе с v-if + v-else, если они являются одним и тем же типом элемента (например, когда оба элемента <div>).

    Vue Style Guide

  • Д7. Не использовать селекторы элементов вместе со scoped

    Воспользуйтесь селекторами классов вместо селекторов элементов в стилях с атрибутом scoped, потому что большое количество селекторов элементов отрабатывает медленно.

    Vue Style Guide

  • Д8. Коммуникация между родительским и дочерними компонентами должна быть явной

    Входные параметры и события должны быть предпочтительным способом коммуникации между родительским и дочерними компонентами вместо использования this.$parent или изменения входных параметров.

    Vue Style Guide

Мой наставник

Для полноценного прохождения курса вам нужно выбрать наставника.

После выбора вы сможете всегда держать под рукой его контакты (почту и ссылки на профиль в соцсетях, которые он укажет для связи со своими студентами), а также расписание консультаций.

Вы в любой момент можете поменять наставника, если вам кажется, что с текущим наладить работу не получается.

Мои проекты

Для удобства работы с проектами, мы собрали их в этом блоке.

Основной личный проект, который вы будете готовить к финальной защите, находится в начале списка и рядом с ним стоит иконка . Если на курсе используется учебный проект, то он будет следующим и с иконкой  . Далее будут идти все остальные проекты, если для них были созданы репозитории.

Вы можете выбрать, какой именно проект будет основным, в любой момент прохождения курса, но мы советуем определиться с выбором как можно скорее.

Оцените наставника

Мы всегда стремимся улучшать качество услуг. И нам важно знать, как идёт ваша работа с наставником.

Уделите пару секунд вашего времени и дайте нам знать, как обстоят дела.

Также вы всегда можете задать свои вопросы напрямую куратору.

Ваши первые задачи

В этом списке мы собрали задачи, с которых начинается обучение на курсе.

Выполняйте задачи по порядку, а система будет автоматически отмечать галочками ваш прогресс. Когда весь список будет выполнен, этот блок закроется и больше не будет вам надоедать.

Стандартный тариф

Студенты стандартного тарифа работают с наставником в течение первых пяти недель обучения. Оставшееся время консультаций, можно потратить на подготовку проекта к защите.

Запись на консультацию

Голосовая консультация — это живое общение с наставником, который ответит на интересующие вас вопросы.

Текстовая консультация — это общение с наставником в чате. Чат доступен 1 день с момента начала консультации.

Кодревью — это проверка пулреквеста в репозитории вашего проекта. Наставник оставит комментарии к вашему коду.

От выбранного раздела зависит, кто из наставников сможет провести консультацию.

Вы можете отправить заявку на консультацию конкретному наставнику, или её возьмёт первый, освободившийся из живой очереди.
